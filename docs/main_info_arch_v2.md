# 流浪动物透明账本小程序 IA v2

## 0. 这版 IA 要先解决什么

这不是一个“惨状信息流”产品。

它首先是一个“救助优先级判断工具”，其次才是一个“透明账本”和“分发工具”。

你最近在救助群里看到的问题很真实：信息很多，情绪很重，但判断结构很弱。大家会看到很多猫狗都很惨，却分不清：

- 谁现在最急
- 谁证据最完整
- 谁已经被支持不少了
- 谁其实只差最后一笔
- 谁还缺关键病历和票据，不适合优先转发

所以 IA 不能只围绕“展示案例”来设计，而要围绕“帮助人做判断”来设计。

这版 IA 的核心改动只有一句话：

**从“浏览案例”改成“判断优先级 -> 建立信任 -> 完成支持 -> 看见闭环”。**

---

## 1. 核心用户与核心任务

### 核心用户

#### A. 个人救助人

- 需要快速建档
- 需要持续更新病情和账本
- 需要把零散聊天记录、票据、截图整理成可信页面
- 需要让外部支持者少问重复问题

#### B. 爱心支持者

- 不是想“逛内容”
- 是想快速判断“这只动物现在是否值得优先帮助”
- 支持后希望看到钱有没有进入对应动物的透明账本
- 希望后续还能看见结果，而不是转完就失联

### 核心任务

#### 支持者的核心任务

在 30-60 秒内判断：

1. 这只动物现在有多急
2. 证据是否足够可信
3. 还差多少钱，差的是哪一段钱
4. 我现在最适合做什么，出钱、转发、还是先观望

#### 救助人的核心任务

在 1-3 分钟内完成：

1. 建档
2. 记账
3. 写更新
4. 审核支持记录
5. 输出可转发的内容

## 2. 顶层信息架构

建议保留 3 个主入口，但重新定义它们的角色。

```text
Tab 1 发现 / 待支持
Tab 2 救助工作台
Tab 3 我的
```

### Tab 1：发现 / 待支持

不是“案例流”，而是“优先级判断入口”。

用户进入这里，不应该先看到一堆差不多的惨图，而应该先看到平台帮他做的第一层结构化判断。

包含 4 个层级：

1. 顶部判断入口
2. 优先队列列表
3. 筛选与解释
4. 个案详情页

### Tab 2：救助工作台

不是普通项目列表，而是“今天该处理什么”的救助控制台。

包含 3 个层级：

1. 今日待办
2. 项目列表
3. 单只动物管理页

### Tab 3：我的

不是占位页，而是“关系沉淀页”。

它的作用不是功能堆砌，而是把“我和这些救助的关系”沉淀下来。

建议只承接 3 件事：

1. 我的支持足迹
2. 我的救助档案
3. 平台规则 / 防骗说明

---

## 3. Tab 1：发现 / 待支持

## 3.1 页面定位

首页要解决的不是“怎么让人看见更多案例”，而是：

**怎么让人快速知道，当前最值得优先支持的是谁。**

## 3.2 页面结构

```text
发现 / 待支持
  A. 今日最需要帮助
  B. 快速筛选条
  C. 待支持列表
  D. 判断说明入口
```

### 模块 A：今日最需要帮助

这是首页最重要的区域。

不要让用户一上来面对平铺案例流。先给一个“平台推荐优先看”的区块，最多 1-3 个卡片。

每张卡片只回答 4 个问题：

- 现在是什么阶段
- 紧急点是什么
- 证据齐不齐
- 目前还差多少

#### 卡片字段建议

- 动物图
- 名字 / 代号
- 当前状态
- 紧急标签
  - 今晚手术
  - 住院观察
  - 已欠费待补
  - 术后恢复关键期
- 证据状态
  - 已有病历 + 票据
  - 有病历待补票据
  - 仅文字说明
- 资金状态
  - 还差 `¥XXX`
  - 已支持 `XX%`
- 最近更新时间
- 主按钮：`先看看为什么推荐这只`

这比直接显示“全部/紧急/进行中/完成”更有用，因为它直接回应用户的判断问题。

### 模块 B：快速筛选条

这里不是频道切换，而是“判断维度切换”。

建议筛选项改成：

- `最急`
- `最缺口`
- `证据最完整`
- `刚更新`
- `快完成`

不建议把“全部 / 紧急 / 进行中 / 完成”作为主筛选。那是系统状态，不是支持者的决策语言。

### 模块 C：待支持列表

列表卡片不需要把所有内容摊开。

一张卡应该帮助用户完成“是否点进去”的决定。

#### 卡片信息顺序建议

1. 动物基础信息
2. 当前紧急原因
3. 证据完成度
4. 资金缺口
5. 最近一次更新摘要

#### 文案示例

- 不推荐：`医疗救助中`
- 推荐：`今晚需要补交住院押金`

- 不推荐：`进行中`
- 推荐：`术后第 2 天，需连续 3 天输液`

#### 卡片底部 CTA

- 主 CTA：`看透明账本`
- 次 CTA：`先转发`

不要在列表卡就放“我要支持”。支持是重决策，应该放到详情页里完成。

### 模块 D：判断说明入口

在首页放一个轻入口：

`我们如何判断“优先支持”`

点击后解释首页排序逻辑，不要装作算法很神秘。

建议明确告诉用户：

- 我们优先展示“紧急且证据较完整”的个案
- 不等于别的动物不值得帮助
- 你也可以按“快完成”“证据完整”来筛选

这会显著提高平台公信力。

## 4. 个案详情页（客态）

## 4.1 页面定位

详情页不是“卖惨页”，而是“做决定页”。

用户点进来以后，必须在很短时间内弄清楚：

1. 这只为什么现在需要帮助
2. 这钱会花到哪
3. 救助人是不是持续在更新
4. 我支持以后会不会看得见结果

## 4.2 页面结构

```text
个案详情页
  A. 当前判断卡
  B. 透明资金看板
  C. 救助人信用背书
  D. 最新进展
  E. 关键时间线
  F. 透明账本与凭证
  G. 支持记录
  H. 底部支持栏
```

### 模块 A：当前判断卡

把“为什么现在该看这只”放到最前面。

字段建议：

- 当前状态
- 一句话风险说明
- 当前最需要解决的问题
- 下一关键节点时间

示例：

- `今晚 22:00 前需补足押金，否则无法继续住院`
- `已完成首诊，但明天要做生化复查`

这个模块解决“急不急”的问题。

### 模块 B：透明资金看板

保留三段式结构，但文案必须白话。

建议从：

- 已获场外支持
- 已核验票据缺口
- 后续预估缺口

改成：

- `已经收到的支持`
- `已经确认花掉、但还没补上的钱`
- `医生判断后面大概率还要花的钱`

右上角继续保留 `?`，但弹窗解释要特别口语化。

### 模块 C：救助人信用背书

必须保留，而且应该更靠前。

在公益和救助场景里，支持者判断的不是“平台厉不厉害”，而是“这个人值不值得信”。

建议字段：

- 救助人昵称 / 头像
- 上次更新时间
- 已发布透明账本数
- 已上传真实凭证数
- 已完成救助数
- 查看主页

### 模块 D：最新进展

把最新一条进展单独提出来，不要让用户自己在时间线里翻。

建议这是详情页里最靠上的内容块之一。

字段建议：

- 最近更新时间
- 最新一句话摘要
- 最新图片 1-3 张
- 状态是否变化

这个模块解决“是不是还在持续救”的问题。

### 模块 E：关键时间线

时间线不要一开始就把所有节点平铺。

建议先展示“关键节点”，再提供“展开全部”。

默认展示：

- 初始发现
- 首次就诊
- 最新支出
- 最新病情更新
- 最近一次支持到账

理由很简单：用户是来做判断的，不是来读长篇日志的。

### 模块 F：透明账本与凭证

这里才是“硬证据区”。

建议结构：

- 最新 3 笔支出
- 查看全部支出
- 每笔支出的对应凭证缩略图
- 是否已核验

这会比把凭证散落在时间线里更容易看懂。

### 模块 G：支持记录

这里的作用不是凑热闹，而是建立“别人也在真实参与”的社会证明。

建议展示：

- 已确认支持人数
- 最近确认的几笔支持
- 脱敏昵称 + 金额 + 留言

### 模块 H：底部支持栏

这里一定要统一路径。

不建议同时并列：

- 我要支持
- 认领支持

这样会让第一次来的用户懵掉。

建议改成：

- 主按钮：`我要支持`
- 次按钮：`帮扩散`

点击 `我要支持` 后进入半屏流程：

1. 选择支持方式
   - 联系救助人
   - 直接支持
2. 完成支持后
   - `我已支持，去认领记录`

也就是说，**认领不是平级入口，而是支持后的下一步。**

## 5. Tab 2：救助工作台

## 5.1 页面定位

工作台不是“我的项目列表”，而是“我的救助待办中心”。

救助人的脑子很乱，时间也很碎。IA 必须优先帮他判断“我现在先处理哪件事”。

## 5.2 页面结构

```text
救助工作台
  A. 今日待办
  B. 快捷动作
  C. 进行中的项目
  D. 草稿箱
```

### 模块 A：今日待办

这是工作台首页最该新增的部分。

字段建议：

- 待确认认领 `3`
- 待补票据 `2`
- 超过 48 小时未更新 `1`
- 预算已调整待说明 `1`

每一条都可以点进去。

这会比单纯显示“2 个进行中项目、1 个草稿”更符合真实工作流。

### 模块 B：快捷动作

首页直接给 4 个高频动作：

- 新建救助档案
- 记一笔支出
- 写进展更新
- 确认支持记录

而不是把所有动作都藏到单只动物详情页后面。

### 模块 C：进行中的项目

项目列表保留，但每张卡不只显示状态，还要显示“下一步动作”。

例如：

- `住院中 · 待补昨晚票据`
- `康复观察中 · 48 小时未更新`
- `已支持 82% · 还差最后 ¥300`

### 模块 D：草稿箱

保留当前设计即可，但建议增加草稿缺失提示：

- 未上传封面
- 未填写预算
- 还未生成公开页

---

## 6. 单只动物管理页（主态）

## 6.1 页面定位

这里不是“详情页的管理版”，而是“单项目控制台”。

要让救助人一眼看出：

- 现在的资金状态
- 最近一次更新是什么
- 还有哪些待处理动作

## 6.2 页面结构

```text
单只动物管理页
  A. 项目总览
  B. 待处理提醒
  C. 高频动作
  D. 最近动态
  E. 完整时间线
```

### 模块 A：项目总览

保留当前的标题、状态、目标金额、三段式进度条。

### 模块 B：待处理提醒

新增一个窄卡片区域：

- 有 2 条待确认认领
- 距上次更新已 3 天
- 最近一笔支出尚未补图

### 模块 C：高频动作

动作顺序建议按真实频率排：

1. 记一笔支出
2. 写进展更新
3. 确认场外收入
4. 调整预算
5. 生成文案

### 模块 D：最近动态

先把最近一条关键动态独立展示，再往下才是全量时间线。

这样救助人打开页面时，不用重新扫描整条历史。

## 7. 支持与认领闭环

## 7.1 关键原则

支持链路必须清楚区分 3 件事：

1. 决定要不要支持
2. 实际通过外部渠道完成支持
3. 回到平台认领这笔支持

现在很多产品会把这 3 件事揉在一起，结果就是用户不敢点，或者点完不知道下一步。

## 7.2 建议流程

```text
详情页
  -> 我要支持
  -> 选择方式
  -> 外部转账
  -> 返回平台
  -> 认领支持记录
  -> 救助人确认到账
  -> 进度条与时间线更新
```

## 7.3 认领页要解决的问题

认领页不是一个普通表单，它本质上是在做两件事：

- 给救助人减轻对账负担
- 给支持者一个“我的支持真的被记录了”的正反馈

所以认领页一定要突出一句话：

`认领成功后，你的支持会显示在透明账本和进度条里。`

## 8. “我的”页

建议把这页收得很克制。

```text
我的
  A. 我的支持足迹
  B. 我的救助档案
  C. 平台规则与防骗说明
```

### 模块 A：我的支持足迹

这是对支持者最重要的关系沉淀。

可以展示：

- 我支持过哪些动物
- 哪些已经被确认到账
- 最近一次更新是什么

### 模块 B：我的救助档案

对救助人来说，这里只是工作台的补入口，不必重复太多。

### 模块 C：平台规则与防骗说明

公益产品一定要把这块做得比普通内容产品更清楚。

建议放：

- 平台不代收代付
- 如何判断可信个案
- 为什么会有“待确认 / 手工补录 / 存疑”状态

## 9. IA 里的关键排序逻辑

如果你真的想解决群里“看不出先帮谁”的问题，平台必须敢于给出结构化判断。

建议每个个案至少有 4 个对外可见判断维度：

### 1. 紧急程度

- 今晚必须处理
- 24 小时内关键节点
- 本周持续治疗
- 可持续关注

### 2. 证据完成度

- 有病历 + 有票据 + 有更新
- 有病历 + 待补票据
- 仅文字说明

### 3. 资金状态

- 刚开档，基本无人支持
- 已有部分支持但仍有较大缺口
- 接近完成，只差最后一段

### 4. 更新活跃度

- 24 小时内更新
- 3 天内更新
- 超过 3 天未更新

首页排序不一定公开算法分数，但可以把这些判断翻译成用户能看懂的标签。

## 10. 这一版 IA 最值得马上采纳的 5 个改动

### 现在就该改

1. 发现页从“分类浏览”改成“优先支持判断入口”
2. 详情页最上方增加“当前为什么急”的判断卡
3. 把“认领支持”从平级按钮改成“支持后的下一步”
4. 工作台首页增加“今日待办”
5. 公开页把账本文案改成白话解释

### 可以先记在 IA / 设计层，暂不实现

1. 首页增加“我们如何判断优先支持”的解释页
2. 时间线默认只展示关键节点，减少阅读负担
3. “我的支持足迹”作为支持者关系沉淀页优先级上升

## 11. 一句话总结

你这个产品真正有机会打动人的地方，不是“比群聊更惨”，而是“比群聊更能帮助人做判断”。

群聊提供情绪。

这个小程序要提供结构、证据、优先级和闭环。

这才是它该赢的地方。
