# 个案详情页 IA

## 1. 页面定位

个案详情页是：

**传播落地页 + 支持决策页 + 信任建立页。**

用户打开后，必须很快弄清：

1. 我是不是进对了这个案例
1. 这只现在为什么需要帮助
2. 钱已经花到哪了
3. 救助人是否持续在更新
4. 我该联系、登记支持，还是帮扩散
5. 如果我支持过不止一次，这些记录会不会被系统正确串起来

---

## 2. 页面结构

```text
个案详情页
  A. 案例 ID 条
  B. 当前判断卡
  C. 透明资金看板
  D. 救助人信用信息
  E. 最新进展
  F. 关键时间线
  G. 透明账本
  H. 支持记录
  I. 底部操作栏
```

### 模块 A：案例 ID 条

在详情页顶部靠前位置展示一条案例 ID 信息：

- `案例 ID：JM482731`
- `复制` 按钮
- `info` 按钮

#### 作用

- 让用户确认“我进的是对的档案”
- 方便在群聊、短视频评论区、其他平台里手打传播
- 形成“看外部内容 -> 搜案例 ID -> 进入透明账本”的查档闭环

#### 按钮行为

##### `复制`

- 复制完整案例 ID
- 默认复制带前缀版本，如 `JM482731`

##### `info`

- 打开一个轻说明弹层
- 文案建议：
  - `这个案例 ID 可以在首页搜索，帮助别人快速找到同一个透明账本。`

### 模块 B：当前判断卡

字段：

- 当前阶段
- 一句话风险说明
- 当前最需要解决的问题
- 下一关键节点时间

这是最前面的判断位。

### 模块 C：资金信息区

一期不再把三段式分段展示作为核心表达。

这里直接展示 4 行核心信息：

- 总预算
- 已确认支出
- 已确认支持
- 当前状态提示

#### 当前状态提示规则

- 如果 `confirmed_support_amount >= confirmed_expense_amount`
  - 显示：`当前垫付已覆盖`
- 如果 `0 < confirmed_expense_amount - confirmed_support_amount <= 2000`
  - 显示：`即将筹满`
- 如果 `confirmed_expense_amount - confirmed_support_amount > 2000`
  - 显示：`‼️ 救助人垫付较多`

并补一句辅助说明：

`总预算代表整体治疗预期，不等于当前立刻需要补齐的金额。`

### 模块 D：救助人信用信息

字段：

- 救助人昵称 / 头像
- 最近更新时间
- 已发布个案数
- 已上传支出记录数
- 已确认支持登记数
- 已完成救助数
- 查看主页

#### 统计口径补充

一期允许把“支持其他救助人的次数”作为主页统计的一部分展示。

它也属于一种真实参与记录，可以作为信任补充。

#### 一期主页形态

救助人主页只做极简版：

- 昵称
- 基础认证状态
- 公开案例卡片列表

不做复杂介绍页。

### 模块 E：最新进展

展示：

- 最近更新时间
- 卡片摘要短句
- 最新图片 1-3 张
- 是否有状态变化

### 模块 F：关键时间线

默认只展示关键节点：

- 初始发现
- 首次就诊 / 首次处理
- 最新支出
- 最新病情更新
- 最近一次支持登记或确认

再提供 `展开全部`。

### 模块 G：透明账本

默认展示：

- 最新 3 笔支出
- 类别
- 金额
- 一句话说明
- 凭证类型标签
- 证据完整度
- 查看全部

### 模块 H：支持记录

这块的目标不是做热闹榜单，而是让支持记录闭环可见。

建议默认展示：

- 已确认支持金额：¥X
- 待处理支持登记：N 条

点击展开后，再按“支持串”查看明细。

#### 支持串展示方式

每个支持串显示：

- 支持者昵称
- 累计已确认金额
- 待处理条数
- 最新一条记录时间

展开后显示该串下的支持记录：

- 时间
- 金额
- 状态
- 留言（可选）
- 是否有截图
- 若为未匹配，显示原因

#### 状态文案

- `pending` -> `待处理`
- `confirmed` -> `已确认`
- `unmatched` -> `未匹配`

这里不要用“驳回”作为前台状态词。

#### 展示原则

- 未匹配记录保留可见
- 未匹配不计入已确认支持金额
- 重复提交通过“未匹配原因 = 疑似重复提交”表达
- 同一个人多次支持时，不拆成多个独立支持者展示

### 模块 I：底部操作栏

一期默认不前置公开付款入口。

按钮建议：

- 主按钮：我要支持
- 次按钮：我已支持
- 辅助按钮：帮扩散

辅助说明：

`如果你已经通过其他方式支持过，可以回到这里登记，等待救助人确认。`

说明：

- 主按钮文案保留“我要支持”
- 但一期不把它做成平台内支付
- 进入后优先展示联系信息与登记闭环
- 半弹层内容区可滚动，但底部主操作需要固定在可点击区域；救助人联系方式允许只提供二维码或只提供微信号，页面按真实可用渠道展示，不补假图占位。

#### “我已支持”提交流程

支持者点击后进入表单：

- 金额（必填）
- 支持时间（必填）
- 留言（可选）
- 支持凭证截图（可选，但引导上传）

提交逻辑：

1. 查找 `(animal_id, supporter_user_id)` 对应支持串
2. 若不存在，则创建支持串
3. 在该串下新增一条支持记录，状态为 `待处理`

#### 一期反刷限制

- 同一用户对同一动物，10 分钟内最多新增 1 条支持记录
- 同一用户对同一动物，24 小时内最多新增 3 条支持记录
- 同一支持串内，相同截图不能重复提交

## 3. 关键文案边界

### 由救助人填写

- 最新情况摘要
- 长文本进展说明
- 本次支出用途说明

### 由系统计算

- 资金状态
- 当前缺口
- 证据完整度
- 推荐理由

这两类信息在 UI 中必须分开展示。
