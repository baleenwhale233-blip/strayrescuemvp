# 个案详情页 IA

## 1. 页面定位

个案详情页不是展示页这么简单。用户从分享、首页或案例 ID 搜索进来后，主要在这里判断下一步。

用户打开后，要很快弄清：

1. 我是不是进对了这个案例
2. 这只现在为什么需要帮助
3. 钱已经花到哪了
4. 救助人是否持续在更新
5. 我该联系、登记支持，还是帮扩散
6. 如果我支持过不止一次，这些记录会不会被系统正确串起来

### 入口身份规则

- 所有已发布详情入口统一只携带 `id`，不再通过路由参数指定主态或客态。
- 详情页先尝试读取当前账号的 owner 详情；如果后端确认当前账号就是记录维护者，展示主态。
- 如果 owner 详情不可用或后端返回无权限，则展示客态，保持外部查看者查档路径稳定。
- 路由参数只允许表达页面位置，例如 `tab=detail`；账号身份一律由后端 owner 权限判断。

## 2. 页面结构

```text
个案详情页
  A. 案例 ID 条
  B. 当前判断卡
  C. 透明资金看板
  D. 救助人信用信息
  E. 最新进展
  F. 关键时间线
  G. 透明账本
  H. 支持记录
  I. 底部操作栏
```

### 模块 A：案例 ID 条

在详情页顶部靠前位置展示一条案例 ID 信息：

- `案例 ID：JM482731`
- `复制` 按钮
- `info` 按钮

#### 为什么放这里

- 让用户确认“我进的是对的档案”
- 方便在群聊、短视频评论区、其他平台里手打传播
- 形成“看外部内容 -> 搜案例 ID -> 进入透明账本”的查档路径

#### 按钮行为

##### `复制`

- 复制完整案例 ID
- 默认复制带前缀版本，如 `JM482731`

##### `info`

- 打开一个轻说明弹层
- 文案建议：
  - `这个案例 ID 可以在首页搜索，帮助别人快速找到同一个透明账本。`

#### 分享直达返回

如果用户从微信分享卡片直接进入详情页，当前页面可能没有上一层页面栈。

这时左上角返回不应停留失效，而应回到 `发现` tab，让用户可以继续通过首页列表或案例 ID 查档。

### 模块 B：当前判断卡

字段：

- 当前阶段
- 一句话风险说明
- 当前最需要解决的问题
- 下一关键节点时间

这是最前面的判断位。

### 模块 C：资金信息区

一期不再把三段式分段图当成核心表达。

这里直接展示 4 行核心信息：

- 总预算
- 已确认支出
- 已确认支持
- 当前状态提示

#### 当前状态提示规则

这里的 `target_amount` 指预算调整后的当前总预算；资金提示、当前缺口和进度条都必须用同一个当前总预算。

- 如果 `target_amount <= 0`
  - 显示：`预算待确认`
- 如果 `confirmed_support_amount >= target_amount`
  - 显示：`已覆盖预算`
- 如果 `0 < target_amount - confirmed_support_amount <= 300`
  - 显示：`即将筹满`
- 其他情况
  - 显示：`当前缺口 ¥{budget_gap_amount}`

并补一句辅助说明：

`总预算代表整体治疗预期，不等于当前立刻需要补齐的金额。`

#### 资金进度条口径

详情页资金条总长度直接代表 `target_amount / 总预算`。

- 已确认支持是唯一填充段
- 当前缺口 = `max(target_amount - confirmed_support_amount, 0)`
- 已确认支出只在数字指标里表达，不参与筹满状态判断
- 不再把已确认支出画进进度条

### 模块 D：救助人信用信息

字段：

- 救助人昵称 / 头像
- 最近更新时间
- 已发布个案数
- 已上传支出记录数
- 已确认支持登记数
- 已完成救助数
- 查看主页

#### 统计口径补充

一期允许把“支持其他救助人的次数”作为主页统计的一部分展示。

它也属于一种真实参与记录，可以作为信任补充。

#### 一期主页形态

救助人主页只做极简版：

- 昵称
- 基础认证状态
- 公开案例卡片列表

不做复杂介绍页。

### 模块 E：最新进展

展示：

- 最近更新时间
- 卡片摘要短句
- 最新图片 1-3 张
- 是否有状态变化

### 模块 F：关键时间线

默认只展示关键节点：

- 初始发现
- 首次就诊 / 首次处理
- 最新支出
- 最新病情更新
- 最近一次支持登记或确认

再提供 `展开全部`。

### 模块 G：透明账本

默认展示：

- 最新 3 笔支出
- 类别
- 金额
- 一句话说明
- 凭证类型标签
- 记录凭证情况
- 查看全部

### 模块 H：支持记录

这块不是榜单，只是让支持记录的处理结果看得见。

建议默认展示：

- 已确认支持金额：¥X
- 待处理支持登记：N 条

点击展开后，再按“支持串”查看明细。

#### 支持串展示方式

每个支持串显示：

- 支持者昵称
- 累计已确认金额
- 待处理条数
- 最新一条记录时间

展开后显示该串下的支持记录：

- 时间
- 金额
- 状态
- 留言（可选）
- 是否有截图
- 若为未匹配，显示原因

#### 状态文案

- `pending` -> `待处理`
- `confirmed` -> `已确认`
- `unmatched` -> `未匹配`

这里不要用“驳回”作为前台状态词。

#### 展示原则

- 未匹配记录保留可见
- 未匹配不计入已确认支持金额
- 重复提交通过“未匹配原因 = 疑似重复提交”表达
- 同一个人多次支持时，不拆成多个独立支持者展示

### 模块 I：底部操作栏

一期默认不前置公开付款入口。

按钮建议：

- 主按钮：查看联系方式
- 次按钮：登记支持
- 辅助按钮：帮扩散

辅助说明：

`如果你已经通过其他方式支持过，可以回到这里登记，等待救助人确认。`

说明：

- 主按钮文案改成“查看联系方式”
- 一期不把它做成平台内支付
- 进入后优先展示联系信息与登记闭环
- 半弹层内容区可滚动，但底部主操作需要固定在可点击区域；救助人联系方式允许只提供二维码或只提供微信号，页面按真实可用渠道展示，不补假图占位。

#### 主态底部操作栏

记录维护者查看自己的档案时，底部默认不直接放大红色结束滑块，避免把低频危险动作做成主操作。

默认态：

- 主按钮：`分享档案`，橙色大按钮，调用小程序分享
- 次按钮：`结束`，小按钮，进入结束确认态

结束确认态：

- 左侧小按钮变为：`取消`
- 主区域显示红色滑块：`右滑结束记录`
- 说明文案：`确认这条记录已完成或已结案时，请滑动结束项目`

原则：

- 分享是记录维护者更高频的正向动作，默认给最高权重
- 结束记录是低频、强风险动作，只在用户主动点击 `结束` 后出现
- 滑动完成后仍需二次确认；当前正式关闭后端链路仍待接入

#### “登记支持”提交流程

支持者点击后进入表单：

- 金额（必填）
- 支持者称呼（必填）
- 支持凭证截图（必填）
- 留言（可选）

提交逻辑：

1. 查找 `(animal_id, supporter_user_id)` 对应支持串
2. 若不存在，则创建支持串
3. 在该串下新增一条支持记录，状态为 `待处理`

#### 一期反刷限制

- 同一用户对同一动物，10 分钟内最多新增 1 条支持记录
- 同一用户对同一动物，24 小时内最多新增 3 条支持记录
- 同一支持串内，相同截图不能重复提交

## 3. 关键文案边界

### 由救助人填写

- 最新情况摘要
- 长文本进展说明
- 本次支出用途说明

### 由系统计算

- 资金状态
- 当前缺口
- 记录凭证情况
- 推荐理由

这两类信息在 UI 中必须分开展示。
