# 我的 / 联系方式设置 / 支持足迹 页面 IA

## 1. 页面定位

这一组页面一期的目标不是做“完整个人中心”，而是承接 3 件事：

1. 显示当前用户的基础身份信息
2. 让用户查看自己的支持足迹
3. 让需要发起救助的人设置联系信息

也就是说，这里更像：

**轻量个人入口 + 支持足迹 + 联系方式配置**

而不是：

**功能复杂的账户中心**

---

## 2. 我的页

## 2.1 页面定位

“我的”页是一个入口页，不承载复杂操作。

它只负责：

- 展示头像和用户名
- 引导进入支持足迹
- 引导进入联系方式设置
- 引导查看使用说明

## 2.2 页面结构

```text
我的
  A. 用户信息区
  B. 功能列表
```

### 模块 A：用户信息区

字段：

- 微信头像
- 微信用户名

#### 一期原则

- 头像和用户名以用户授权 / 确认后的结果为准
- 不要求复杂编辑
- 没拿到授权时允许占位

### 模块 B：功能列表

保留 3 个入口：

1. 我的支持足迹
2. 联系方式设置
3. 救助账本使用说明

---

## 3. 我的支持足迹页

## 3.1 页面定位

这是支持者视角的轻量回看页。

目标不是做复杂对账，而是让用户知道：

- 我支持过哪些动物
- 我对每只动物累计登记了多少钱
- 点击后可以回到该动物详情页继续看最新情况

## 3.2 进入条件

- 用户提交过至少 1 次“我已支持”后，这个页面才有内容
- 如果没有内容，显示空状态，不隐藏入口

## 3.3 页面结构

```text
我的支持足迹
  A. 总计支持
  B. 支持记录列表
```

### 模块 A：总计支持

字段：

- `总计支持金额`

这是该用户所有已登记支持金额的汇总展示。

### 模块 B：支持记录列表

一期不要做太复杂。

每条记录只展示：

- 动物头像
- 动物名
- 我的累计支持金额

点击后：

- 进入对应动物详情页
- 再看最新情况、资金状态和支持闭环

#### 一期不做

- 不在这里展示 thread / entry 明细
- 不在这里展示复杂状态筛选
- 不在这里展示大量时间线信息

---

## 4. 联系方式设置页

## 4.1 页面定位

这是给“需要发起救助的人”准备的配置页。

目标是让用户以最低摩擦补齐：

- 联系方式
- 联系二维码
- 联系说明

## 4.2 页面结构

```text
救助联系方式
  A. 微信号
  B. 微信二维码
  C. 备注
  D. 提交按钮
```

### 模块 A：微信号

字段：

- `wechat_id`

UI 形态：

- 单行输入框

说明：

- 一期按“用户手填”处理
- 不依赖系统自动获取微信号

### 模块 B：微信二维码

字段：

- `wechat_qr_image`

UI 形态：

- 上传图片入口

说明：

- 一期按“用户手动上传二维码图片”处理
- 不假设小程序可以自动拿到用户个人微信二维码

### 模块 C：备注（选填）

字段：

- `contact_note`

UI 形态：

- 多行文本框

用途：

- 让救助人补充联系说明

示例：

- 添加请备注案例 ID
- 只接受微信联系
- 晚上回复较慢

### 模块 D：提交按钮

按钮：

- `提交`

一期不做复杂多状态保存流程，先以明确提交为主。

---

## 5. 使用说明页

## 5.1 页面定位

这是一个文档页，不是复杂功能页。

主要承接：

- 鼓励实际救助人自己发救助档案，不建议代发二手档案
- 平台不碰资金
- 怎么看透明账本
- 怎么联系救助人
- 怎么登记“我已支持”
- 基础防骗说明

一期不需要拆很多二级帮助页。

正式用户文案见：

- [`docs/rescue_ledger_usage_guide.md`](/Users/yang/Documents/New%20project/stray-rescue-mvp/docs/rescue_ledger_usage_guide.md)

当前页面结构：

```text
救助账本使用说明
  A. 顶部说明
  B. 为什么要实际救助人自己建档
  C. 账本做什么 / 不做什么
  D. 救助人使用步骤
  E. 支持者使用步骤
  F. 好档案标准
  G. 不建议做的事
```
