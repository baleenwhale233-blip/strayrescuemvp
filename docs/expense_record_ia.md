# 记账 / 支出记录 页面 IA

## 1. 页面定位

记账页不是财务报销页，而是：

**低负担透明记录工具。**

一期目标：

- 让救助人愿意记
- 让支持者能看懂
- 让平台能做轻量分层

---

## 2. 页面结构

```text
记账
  A. 关联动物
  B. 固定字段
  C. 凭证区
  D. 记录凭证预览
  E. 提交
```

### 模块 A：关联动物

- 默认带入当前动物
- 不要求用户重复选择

### 模块 B：固定字段

每条支出固定字段：

- 金额
- 日期
- 类别
  - 医疗
  - 药品
  - 食物/用品
  - 交通/其他
- 一句话说明

不要求用户手写自由标题作为主字段。

### 模块 C：凭证区

#### 单条凭证槽位

- 票据 / 订单截图
- 转账 / 支付截图
- 物品照片 / 药品照片
- 治疗相关图
- 猫咪当次照片
- 猫咪与物品同框图

#### 公共凭证

对于一组支出，允许先上传：

- 公共订单图
- 公共支付图
- 公共场景图

再把公共凭证共享给多条明细。

#### 当前交互规则

- 新记账提交时必须至少上传 1 张图片；如果是历史遗留的无图记录，前台按纯文本记录展示，不补假凭证图或占位图
- 当公共凭证缩略图超出单行宽度时，应支持横向滚动查看，不做静态截断
- “添加照片”按钮应固定可见，不随公共凭证横向滚动一起移走
- 公共凭证列表默认按最新上传在前展示，避免用户上传后看不到新增反馈
- 点击已上传缩略图，应能直接查看大图
- 离开页面时应静默缓存当前未提交内容；再次进入同一案例的记账页时，先让用户选择“继续上次录入”或“新的录入”
- 已提交支出允许记录维护者从支出详情进入编辑模式，编辑仍使用同一套“公共凭证 + 多条明细”表单
- 每次支出修改必须保留修改记录：记录原金额 / 原说明 / 原明细与新金额 / 新说明 / 新明细，详情页可回看；不能静默覆盖旧值
- 修改保存成功后，应只返回原支出详情页，并由支出详情重新读取远端记录，立即看到修改后的明细、金额和修改记录；不直接退回个案详情页
- 修改记录按实际修改时间从旧到新展示，编号按展示顺序递增；最新一次修改位于列表下方，并显示当前最高修改次数

### 模块 D：记录凭证预览

系统实时显示：

- 完整
- 基础
- 待补充

并提示这条记录的效果：

- 是否足以公开展示
- 是否仍需补凭证
- 是否会影响首页资格

### 模块 E：提交

主按钮：

- `保存记录`

次按钮：

- `继续添加下一笔`

## 3. 批量记支出

## 3.1 适用场景

- 一次下单买了多种用品
- 一次支付包含多个小额支出
- 一组明细共享同一组凭证

## 3.2 页面结构

```text
批量记支出
  A. 当前动物
  B. 上传公共凭证
  C. 本次合计支出
  D. 添加多条明细
  E. 预览记录凭证情况
  F. 提交
```

## 3.3 本次合计支出

- 合计金额默认放在“支出明细”标题下方，作为进入明细录入前的实时反馈
- 当用户向下滚动、原合计区离开顶部可读区域后，在页面导航下方出现轻量吸顶汇总条
- 吸顶条只展示 `本次合计支出 + 金额`，高度控制在约 44-52px，不做大卡片
- 底部固定按钮保持单一提交动作，避免合计信息和 `确认并挂载至账本` 抢占底部空间

## 3.4 明细字段

每条明细只要求：

- 金额
- 类别
- 一句话说明

公共凭证统一挂在这一组记录上。

---

## 4. 与首页资格的关系

### 可公开的最小要求

- 至少完成基础建档
- 至少有一条可解释的记录

### 可进入首页的推荐要求

- 最近有更新
- 至少有基础支出记录
- 支出记录至少已有基础凭证

### 一期当前执行口径

当前首页准入已经落地的部分，实际只看这几件事：

- 案例已公开
- 至少有 1 条公开进展更新
- 至少有 1 条支出记录，且记录凭证情况不是 `待补充`

“异常 / 重复风险”目前仍属于后续增强方向，还没有进入首页准入的当前执行口径。

所以提示文案要始终是：

`可以先公开，再逐步补到能进首页。`
