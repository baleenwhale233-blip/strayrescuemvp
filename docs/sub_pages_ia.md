# 「写进展更新」二级页面信息架构 (IA)
**页面定位：** 救助人的“高频发声”工具。解绑财务数据，纯粹用于记录生命状态流转和日常更新，以维持捐助者的情绪价值和信任。
# 1. 页面入口与前置数据
* **入口位置：** 救助工作台（主态） -> 单只动物管理面板 -> 点击【写进展更新】按钮。
* **前置携带数据：** 进入页面时，系统静默带入当前动物的 ID 和 当前所处状态。

⠀2. 核心表单模块 (UI 骨架)
### 模块一：救助阶段变更 (核心风控与标签)
* **字段类型：** 必填项 (Required)
* **UI 交互：** 横向滑动的单选标签组 (Chip Group)。
* **默认状态：** 默认选中该动物当前的最新状态。
* **枚举值 (示例)：**
  * 🚨 紧急送医
  * 🏥 医疗救助中
  * 🏡 康复观察中
  * 💖 寻找领养
  * 🌈 遗憾离世
* **说明：** 每次写进展，都强制救助人对当前阶段进行一次确认或变更。这不仅是这次更新的分类，更是整个项目的“定海神针”。

⠀模块二：近况影像记录 (视觉证据)
* **字段类型：** 选填项 (Optional，但 UI 上强烈提示建议上传)
* **UI 交互：** 常见的九宫格图片上传组件。
* **功能限制：**
  * 最多上传 9 张图片。
  * （MVP阶段可暂不考虑视频，或限制在 15 秒短视频以控制服务器成本）。
  * 支持点击大图预览和删除。

⠀模块三：进展详情描述 (内容主体)
* **字段类型：** 必填项 (Required)
* **UI 交互：** 多行文本输入框 (Textarea)。
* **占位提示文案 (Placeholder)：** “例如：大黄今天精神好多了，能主动吃半个罐头，医生说明天可以安排拆线...”
* **字数限制：** 建议 500 - 800 字以内，鼓励短平快的更新，而非长篇大论。

⠀3. 底部操作区 (Actions)
* **悬浮吸底按钮组：**
  * [ 取消 ]：点击后二次弹窗确认“是否放弃本次编辑？”，确认则返回上一页。
  * [ 发布进展更新 ] (主视觉按钮)：点击后触发校验并提交。

⠀4. 提交后的联动逻辑 (系统后台触发事件)
当用户点击【发布进展更新】并成功写入数据库后，系统需立刻触发以下三个动作：
**1** **更新全局状态：** 若【模块一】的状态发生了变更（例如从“医疗救助中”变成了“康复观察中”），系统立即覆盖该动物的全局 Profile。主态首页和客态详情页头部的状态标签瞬间同步改变。
**2** **生成时间线节点：** 在公开页（客态）的瀑布流时间线上，生成一张全新的【状态卡片】。展示时间戳、状态变更提示（如有）、文字描述和带防盗水印的配图。
**3** **分发引导 (Aha Moment 延续)：** 发布成功后，轻量级弹窗提示：“发布成功！要顺便生成一条同步到小红书/微信群的最新文案吗？” -> 导流至 AI 自动化文案分发模块。


# 场外收入与支持登记闭环 - 页面信息架构 (IA)
本架构包含两个深度联动的页面，分别服务于“支持者补登记”和“救助人核实/补录”的场景，目标是在不做平台内支付的前提下，完成场外支持的结构化入账。
# 页面一：【登记我的支持】表单页 (客态 / 支持者视角)
**页面定位：** 支持者在微信私聊、线下转账或其他外部渠道完成支持后，回到小程序进行的“补登记”动作。需要上传凭证供救助人核实。
### 1. 顶部：救助标的确认 (防填错)
* **UI 展现：** 一张极简的动物信息卡片（正脸头像 + 名字，如“瘸腿狸花”）。
* **文案提示：** “感谢你的支持。请补登记本次支持，救助人核实后会展示在进度条和透明账本中。”

⠀2. 核心表单区
* **转账截图 / 凭证 (核心证据)：**
  * **类型：** 必填项 (Required)
  * **UI：** 图片上传组件（单张即可）。
  * **提示：** “请上传微信/支付宝转账成功截图，仅救助人可见，用于核对账单。”
* **支持金额：**
  * **类型：** 必填项 (Required)
  * **UI：** 大字号数字输入框，带“¥”符号。
* **您的称呼 (支持者昵称)：**
  * **类型：** 必填项 (Required)
  * **UI：** 文本框（默认带入微信授权昵称，允许用户修改，如“爱心人士王女士”）。
* **爱心留言 / 备注：**
  * **类型：** 选填项 (Optional)
  * **UI：** 多行文本框。
  * **提示：** “给毛孩子留句祝福吧，或者备注资金的具体用途（如：买罐头专用）。”

⠀3. 底部操作区
* **按钮：** [ 提交登记 ]
* **提交后逻辑：** 1. 生成一条状态为【待确认】的支持记录。 2. 页面跳转回客态详情页，并弹窗提示：“已通知救助人，核实后进度条会自动更新。” 3. 救助人工作台收到小红点提示。

⠀页面二：【记场外收入】管理页 (主态 / 救助人视角)
**页面定位：** 救助人的私人账房。支持“核实支持者补登记”和“无凭证极速手动补录”。
### 1. 页面布局：双 Tab 切换结构
* **Tab A：**待核实支持登记 (带红点数字角标)
* **Tab B：**手动记一笔

⠀2. Tab A：待核实支持登记 (处理别人的上报)
* **列表呈现：** 瀑布流展示所有待处理的支持登记卡片。
* **单张卡片信息：**
  * 支持者称呼与留言（如：王女士 - “给大黄加个鸡腿”）。
  * 申报金额（高亮显示，如 ¥ 50.00）。
  * 凭证缩略图（**点击可放大查看**，救助人据此去微信余额里核对）。
* **卡片操作区：**
  * [ 驳回 ]：金额、对象、凭证或真实性无法成立。
  * [ 标记重复 ]：这笔支持已经被救助人手工补录，或与其他登记重复。
  * [ 确认 ]：主视觉按钮。点击即视为核实完成。
* **三种结果状态：**
  * 【确认】进入时间线和进度条。
  * 【重复】不重复入账，但保留登记痕迹，向支持者解释“这笔已存在记录”。
  * 【驳回】记录作废，可选填原因。
* **确认后联动：** 该记录状态变为【确认】，**顶部的“已筹集”进度条瞬间上涨**，时间线自动生成一条绿色的收入动态卡片。

⠀3. Tab B：手动记一笔 (救助人自己记账)
**场景：** 长辈转账、群里没用小程序的熟人发了红包。**极简录入，无需自己传截图自证。**
* **表单信息：**
  * **支持金额：** 必填项，数字键盘录入。
  * **支持者称呼：** 必填项（如“微信群友-李哥”）。
  * **备注：** 选填项（如“红包直接转账”）。
  * **[!] 注意：不设“上传凭证”组件。** * **底部按钮：** [ 保存收入记录 ]
* **提交后联动：** 直接生成一条【已确认】的支持记录，**进度条立即上涨**，时间线生成收入卡片。卡片上会带有隐性的标记表明来源为“救助人手工补录”。

⠀架构优势总结
**1** **宽严相济：** 对支持者要求“截图证明”（严），防止恶意刷单捣乱进度条；对救助本人提供“免图补录”（宽），彻底减轻心智负担。
**2** **重复可处理：** 支持者补登记和救助人手工补录在真实场景里确实会重复，所以核实页必须支持【重复】而不是只有“确认/驳回”。
**3** **闭环极短：** 支持者填个数字传张图 -> 救助人点一下“确认”，资金的流转和账本的同步就无缝衔接了。
**4** **情绪正反馈：** 支持者的“留言”被确认后会直接展示在时间线上，成为公开的爱心痕迹，极大地满足了参与感。

# “上传票据与记账”模块详细信息架构设计
本模块位于“救助工作台”的单只动物管理面板中，是救助人更新账本最核心的路径。核心设计原则：**AI 承担脏活累活，人类只做最后确认。**
# 1. 模块核心流程 (User Flow)
```
    A[工作台: 记一笔支出] --> B{选择获取方式}
    B -->|拍照| C[相机页: 票据对齐引导]
    B -->|相册| D[相册选择]
    C --> E[AI 异步解析中 Loading]
    D --> E
    E --> F[结构化数据确认页]
    F --> G{数据是否准确?}
    G -->|需要修改| H[手动编辑异常字段]
    G -->|准确无误| I[补充阶段标签/备注]
    H --> I
    I --> J[保存并上链/记入账本]
    J --> K[触发: 是否一键生成求助文案?]
```

# 2. 页面与组件层级架构
### 2.1 拍摄/上传页 (Capture Page)
**目标：获取高质量的原始凭证，降低后续 OCR 难度。**
* **全屏相机视图 (优先调用)**
  * **UI 引导层：** 屏幕中央显示一个半透明的长条形“取景框”（针对医院常见的长条热敏小票设计）。
  * **文案提示：** “请将小票边缘对齐虚线，确保金额和医院名称清晰”。
* **底部操作区**
  * 拍照快门大按钮。
  * [入口] 从相册导入截图（兼容微信群里医生发来的电子账单截图）。
  * [入口] 纯手动记账（极小字号的兜底入口，仅在无可拍照凭证时使用，系统会标记为“无凭证支出”）。

⠀2.2 AI 智能解析过渡页 (Loading State)
**目标：缓解等待焦虑，建立“专业、智能”的心理预期（极其重要的交互细节）。**
* **视觉呈现：** 刚才拍摄的票据图片，上方有类似扫描线的动画扫过。
* **动态文案反馈 (骨架屏思路)：** * 0-1秒：“正在识别医院印章与时间...”
  * 1-2秒：“正在提取核心诊疗项目...”
  * 2-3秒：“正在核对总计金额...”
  * 3秒+：“即将完成，请准备核对。”

⠀2.3 核心：数据校对与确认页 (Data Verification Page)
**目标：展示 AI 成果，允许快速纠错，并补全业务上下文。**
此页面分为上下/左右结构（移动端建议上下滑动，上方固定图片，下方表单）。
* **顶部区：原始凭证对照**
  * **图片缩略图：** 刚才上传的票据（支持点击放大、双指缩放，方便用户对照表单里的数字）。
* **核心表单区：AI 提取结果 (高亮展示)**
  * **金额总计 (必填/最醒目)：** 大字号输入框。AI 置信度低时（如把 ¥100.00 识别成 10000），用橙色高亮提示用户“请核对金额”。
  * **就诊日期 (必填)：** 默认带出小票上的时间，不可为空。
  * **医院名称 (选填但推荐)：** 用于后续风控交叉验证。
  * **核心明细折叠面板 (选填)：** * AI 自动过滤挂号费、留置针等琐碎耗材。
    * 仅列出单项金额 > 50 元的核心大项（如：生化全套 ¥350、住院费 ¥100）。
    * 允许用户左滑删除或手动修改。
* **业务上下文区：关联救助阶段**
  * **阶段标签 (单选)：** 这一笔钱花在什么阶段？（选项：初步检查、住院治疗、急救手术、复查买药、日常粮砂）。*此标签直接决定了这笔钱在“透明公开页”时间线上的归属。*
  * **简短备注 (选填)：** 默认带入 AI 总结的几句话（例如：“本次主要花费为绝育手术及术后消炎”）。
* **底部操作区**
  * **主按钮：** 【确认无误，记入账本】。
  * **次按钮：** 【重新拍摄】。

⠀3. 异常与边界场景处理 (Edge Cases)
| **异常场景** | **AI/系统表现** | **交互兜底方案** |
|---|---|---|
| **票据过于模糊/揉搓破损** | OCR 无法提取有效金额 | 弹窗提示：“未检测到清晰的金额数据”。自动跳转至带有原图的**全手动填表页**。 |
| **金额逻辑错误** | 提取的明细相加 ≠ 总金额 | 顶部横幅警告：“明细总和与总金额不符，请人工确认”。但**允许用户强制保存**（救助现场混乱，不可强卡流程）。 |
| **上传了非票据图片 (如猫咪照片)** | OCR 提取不到任何财务要素 | 阻断并提示：“这似乎不是一张收费凭证。是否改为‘更新病情进展’？”（智能路由，避免用户重头操作）。 |
| **单张小票极其长 (超过单屏)** | 相机无法一次拍全 | （Phase 2 考虑增加长图拼接能力）MVP 阶段引导用户分段拍摄，或提示“尽量拍到总金额与医院盖章部分”。 |
# 4. 数据写入后的联动 (Post-Action)
用户点击“确认记入账本”后，系统后台触发以下操作：
**1** **账本更新：** 对应救助档案的“已花费金额”增加，进度条更新。
**2** **时间线更新：** 在对外公开页的“数字病历”时间轴上，新增一个节点，挂载该票据（自动打上“已核验”平台水印防盗用）。
**3** **转化引导 (Aha 延伸)：** 弹出一个轻量提示框：“账本已更新！是否用新的进度条生成求助海报发群？” -> 导流到 AI 文案分发模块。
