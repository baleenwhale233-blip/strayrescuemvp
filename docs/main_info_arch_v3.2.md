# 流浪动物透明账本小程序 IA v3.2

## 0. 这版 v3.2 在修什么

v3.1 已经把产品收束成了更接近真实救助场景的结构：

- 首页克制，不做惨状信息流
- 个案页做强，承担传播和转化
- 记账走统一支出模板
- 支持与登记解耦
- 首页准入与公开转发分层

但结合 `gstack` review，这一版还要再补 4 个关键缺口：

1. **首页卡片还缺“为什么现在先看这只”的推荐理由位**
2. **首页资格状态还不够白话，容易让救助人误解为“没进首页 = 不可信”**
3. **统一支出模板还需要把“公共证据”提升成主机制，降低真实记录负担**
4. **支持登记闭环需要正式收成三态：驳回 / 重复 / 确认**

一句话总结：

**v3.2 不改大方向，重点把 v3.1 从“方向正确”推进到“页面和状态都能落地”。**

## 1. 核心产品原则

### 原则 1：首页是判断入口，不是内容广场

首页的任务不是让用户“多看几个可怜个案”，而是让用户在极短时间里判断：

- 现在优先看谁
- 为什么优先看它
- 它目前还差什么
- 我下一步应该做什么

所以首页卡片固定展示 3 类信息：

1. **最新情况**，由救助人填写
2. **资金状态**，由系统计算
3. **推荐理由**，由系统输出

### 原则 2：公开转发和首页展示继续拆开

#### A. 可公开转发

达到基础建档条件即可：

- 可生成公开页
- 可在群聊、朋友圈、小红书等外部渠道传播

#### B. 可进入首页

只有满足首页条件的个案，才进入首页列表。

这里要非常明确：

**没进首页，不等于不可信，也不等于不能传播。**

### 原则 3：外部平台导流优先走“查档”，不是强留联系方式

很多外部平台不适合：

- 直接留联系方式
- 长篇解释个案背景
- 直接贴收款方式

所以一期增加：

**可搜索的案例 ID。**

它的作用不是替代首页，而是让外部传播变成：

```text
看到短文案 / 短视频 / 图片
  -> 记住案例 ID
  -> 回小程序搜索案例 ID
  -> 进入透明账本详情页
```

这样既克制，也更适合跨平台导流。

### 原则 4：支出记录做“低负担透明”，不做“强审计”

一期先保证：

- 救助人愿意记
- 支持者看得懂
- 平台能做轻分层

不要求每条记录都像财务报销单。

### 原则 5：支持和登记支持彻底解耦

真实支持会发生在平台外：

- 群聊转账
- 私聊沟通
- 线下支持

所以平台不假设“点按钮就代表已付款”，平台只负责：

- 联系入口
- 登记入口
- 救助人核实
- 闭环更新

### 原则 6：支持登记要可追溯、不可删除、可核实

一期不再把“我已支持”理解成一条孤立记录，而是理解成：

**同一支持者对同一只动物的一条支持关系串。**

也就是：

- 一个支持者对一只动物只有 1 个支持登记串
- 这个串下面可以有多条支持记录
- 记录不能删除
- 救助人只能核实或标记未匹配

这样才能同时处理：

- 同一个人分多次支持
- 支持者补登记和救助人手工补录会重复
- 救助人一直不处理或故意不处理的风险

### 原则 7：状态和原因分开

一期的顶层状态保持 3 个：

- `待处理`
- `已确认`
- `未匹配`

其中“重复”不作为独立顶层状态，而作为 `未匹配` 的原因之一。

这样更稳，因为“重复”本质上是：

**当前这条不应该再次入账**

它不等于用户造假，也不等于系统判定双方谁在撒谎。

## 3. Tab 1：待支持

## 3.1 页面定位

首页不是在回答“还有哪些个案”，而是在回答：

**为什么这只值得你现在先点进去。**

## 3.2 页面结构

```text
待支持
  A. 顶部说明、搜索与筛选
  B. 待支持列表
  C. 判断说明入口
```

### 模块 A：顶部说明、搜索与筛选

页面开头用一句很轻的说明把预期定住：

- 首页展示的是符合平台首页条件的个案
- 不是所有公开个案都会在这里出现
- 你也可以通过筛选切换判断维度

#### 搜索入口

首页顶部增加一个轻量搜索入口，只做：

**案例 ID 精确搜索**

支持输入格式：

- 纯数字，如 `482731`
- 带前缀，如 `JM482731`

行为：

- 精确命中后，直接跳个案详情页
- 搜不到时，只给轻提示，不做复杂结果页

一期不做：

- 动物名字搜索
- 模糊搜索
- 组合筛选搜索

#### 案例 ID 规则

每个公开案例生成一个唯一、可传播、适合手打的短 ID。

建议格式：

- 前台主展示：`JM482731`
- 搜索兼容输入：`482731` / `JM482731`

这个 ID 的作用是跨平台导流，不是内部数据库主键。

#### 主筛选项

- `已垫付较多`
- `证据最完整`
- `刚更新`
- `快完成`

这里不保留 `最缺口`。

原因是“一期的缺口”里混有预算成分，而预算天然比已确认垫付更虚。首页不应让一个刚写下的大预算天然获得更高展示权重。

#### 辅助筛选项

- 医疗中
- 康复中
- 待安置
- 已完成

辅助筛选放进二级区域，不抢主判断位。

### 模块 B：待支持列表

一期不单独做“平台推荐位”。

首页直接展示：

**所有有资格进入首页的案例卡片，并按最近更新时间倒序排序。**

每张卡固定展示：

- 动物名 / 代号
- 当前阶段
- 最新情况
- 资金状态
- 推荐理由
- 证据状态
- 最近更新时间
- 主 CTA：`看透明账本`

#### 资金状态文案规则

首页卡片里的“资金状态”统一用 3 档：

- 如果 `confirmed_support_amount >= confirmed_expense_amount`
  - `当前垫付已覆盖`
- 如果 `0 < confirmed_expense_amount - confirmed_support_amount <= 2000`
  - `即将筹满`
- 如果 `confirmed_expense_amount - confirmed_support_amount > 2000`
  - `‼️ 救助人垫付较多`

这里的状态文案只基于已确认支出和已确认支持，不基于总预算差额。

#### 推荐理由规则

一期固定只做 4 类规则，从上到下命中第一条就显示，不同时展示多条：

1. **垫付压力型**
   - 示例：`已垫付 ¥1800，仍待补位`
2. **刚更新仍缺口型**
   - 示例：`刚更新病情，当前仍缺 ¥420`
3. **接近完成型**
   - 示例：`接近完成，只差最后 ¥300`
4. **证据完整型**
   - 示例：`证据完整，当前仍有缺口`

推荐理由一定是系统判断，不和“最新情况”混在一起。

#### 字段口径

推荐理由一期只基于以下字段：

- 最近一次更新摘要
- 最近公开活动时间
- 已确认支出总额
- 已确认支持总额
- 当前已确认缺口
- 个案证据完整度

详细的字段条件与文案模板映射见：

- [`/Users/yang/Documents/New project/stray-rescue-mvp/docs/home_page_ia.md`](/Users/yang/Documents/New%20project/stray-rescue-mvp/docs/home_page_ia.md)

列表卡片固定信息顺序：

1. 动物基础信息
2. 最新情况
3. 资金状态
4. 推荐理由
5. 证据完整度
6. 最近更新时间

#### 卡片 CTA

- 主 CTA：`看透明账本`
- 次 CTA：`帮扩散`

不在列表卡片直接放“联系救助人”或“我要支持”。

### 模块 C：判断说明入口

保留一个轻入口：

`我们如何把个案放到首页`

解释要点：

- 首页不是按惨烈程度排序
- 也不是按“预算写得多”排序
- 公开转发不等于首页展示
- 平台更看重持续更新、基础证据、当前缺口和闭环可能性

## 4. 个案详情页（客态）

## 4.1 页面定位

个案详情页是传播落地页，也是支持决策页。

它要同时解决 5 件事：

1. 建立信任
2. 解释当前情况
3. 解释钱花到哪
4. 让用户自然进入“联系 / 登记支持 / 帮扩散”
5. 让外部导流用户能确认“我进对档案了”

## 4.2 页面结构

```text
个案详情页
  A. 案例 ID 条
  B. 当前判断卡
  C. 透明资金看板
  D. 救助人信用信息
  E. 最新进展
  F. 关键时间线
  G. 透明账本
  H. 支持记录
  I. 底部操作栏
```

### 模块 A：案例 ID 条

在详情页靠上位置展示：

- 案例 ID
- `复制` 按钮
- `info` 按钮

#### 作用

- 让用户确认自己进对了档案
- 方便在群聊、短视频评论区、其他平台里传播
- 让外部导流不依赖长文说明或直接贴联系方式

#### 按钮行为

- `复制`
  - 复制完整案例 ID，如 `JM482731`
- `info`
  - 轻说明弹层
  - 说明这个 ID 可以在首页搜索，用来快速查到同一个案例

### 模块 B：当前判断卡

放最前面，直接回答“为什么现在该看这只”。

字段：

- 当前阶段
- 一句话风险说明
- 当前最需要解决的问题
- 下一关键节点时间

### 模块 C：资金信息区

一期不再把三段式分段展示作为核心表达。

这里直接展示 4 行核心信息：

- 总预算
- 已确认支出
- 已确认支持
- 当前状态提示

#### 当前状态提示规则

- 如果 `confirmed_support_amount >= confirmed_expense_amount`
  - 显示：`当前垫付已覆盖`
  - 同时保留总预算，补一句提醒：
    - `后续可能仍有预算需求，但当前已确认支出已被覆盖`
- 如果 `0 < confirmed_expense_amount - confirmed_support_amount <= 2000`
  - 显示：`即将筹满`
- 如果 `confirmed_expense_amount - confirmed_support_amount > 2000`
  - 显示：`‼️ 救助人垫付较多`

#### 表达原则

- 总预算保留，用来解释整体治疗预期
- 当前是否需要补位，只看已确认支出与已确认支持
- 不让用户把总预算误读成“现在立刻缺这么多”

### 模块 D：救助人信用信息

字段：

- 救助人昵称 / 头像
- 最近更新时间
- 已发布个案数
- 已上传支出记录数
- 已确认支持登记数
- 已完成救助数
- 查看主页

#### 一期救助人主页

保留一个极简救助人主页，作为信任补充。

只展示：

- 救助人昵称
- 基础认证状态
- 公开案例卡片列表

不做：

- 复杂个人介绍
- 粉丝 / 关注
- 详细历史数据分析
- 多 tab 主页

### 模块 E：最新进展

独立展示最近一次更新：

- 最近更新时间
- 卡片摘要短句
- 最新图片 1-3 张
- 是否有状态变化

### 模块 F：关键时间线

默认展示关键节点：

- 初始发现
- 首次就诊 / 首次处理
- 最新支出
- 最新病情更新
- 最近一次支持登记或确认

再提供 `展开全部`。

### 模块 G：透明账本

默认展示：

- 最新 3 笔支出
- 类别
- 金额
- 一句话说明
- 凭证类型标签
- 证据完整度

### 模块 H：支持记录

作用是建立真实参与感，同时让“我已支持”有可回看的闭环。

建议展示：

- 已确认支持金额
- 待处理支持登记数
- 支持登记入口
- 展开后按支持串展示摘要

#### 一期展示原则

- 默认不把支持区做得很重
- 先展示 `已确认支持金额` 和 `待处理支持登记`
- 用户点击展开后，再查看 thread / entries 明细
- 未匹配记录保留可见，但不计入已确认金额

### 模块 I：底部操作栏

一期默认不前置公开收款码。

建议按钮：

- 主按钮：`联系救助人`
- 次按钮：`我已支持`
- 辅助按钮：`帮扩散`

并补一行辅助说明：

`如果你已经通过其他方式支持过，可以回到这里登记，等待救助人确认。`

## 7. 统一支出模板 v1

目标：

**所有支出尽量走同一个模板，降低记账摩擦。**

### 7.1 一条支出的固定字段

- 关联动物
- 金额
- 日期
- 类别
  - 医疗
  - 药品
  - 食物/用品
  - 交通/其他
- 一句话说明
- 证据

### 7.2 证据采用“通用槽位 + 公共证据”

#### 单条证据槽位

- 票据 / 订单截图
- 转账 / 支付截图
- 物品照片 / 药品照片
- 治疗相关图
- 猫咪当次照片
- 猫咪与物品同框图

#### 公共证据机制

一组支出可以共享公共证据。

例如一次下单：

- 1 张订单截图
- 1 张支付截图
- 1 张猫咪当次照片

下面拆出多条支出明细：

- 猫粮 ¥89
- 尿垫 ¥32
- 益生菌 ¥45

这样记录真实，也不会逼用户给每条明细重复补图。

### 7.3 证据完整度分层

#### 完整

有交易证明 + 有场景证明

#### 基础

只有交易证明，或只有场景证明

#### 待补充

只有文字说明，或证据过少

这三个等级会影响：

- 账本标签
- 首页资格提醒
- 顶部通知状态

### 7.4 公开与首页两层要求

#### 可公开的最小要求

- 至少完成基础建档
- 至少有一条可解释的记录

#### 可进入首页的推荐要求

- 最近有更新
- 至少有基础支出记录
- 支出记录达到基础证据完整度
- 没命中明显异常或重复风险

也就是说：

**可以先公开，再逐步补到能进首页。**

## 8. 支持与登记闭环

## 8.1 基本原则

平台拿不到真实支付状态，所以支持和登记必须解耦。

同时，一期要避免两种坏结果：

1. 救助人一直不确认或错误驳回，导致支持者记录消失
2. 坏人不断刷“我已支持”记录，把个案页面刷得很乱

所以这里的核心不是“单条认领”，而是：

**支持登记串（support thread）+ 支持记录（support entry）**

真实路径是：

```text
看到公开页
  -> 联系救助人
  -> 在线下 / 私聊 / 外部方式支持
  -> 回平台点击“我已支持”
  -> 查找或创建该用户在该动物下的支持登记串
  -> 新增一条支持记录（待处理）
  -> 救助人核实
  -> 支持记录与进度更新
```

## 8.2 文案

- 按钮：`我已支持`
- 页面标题：`登记我的支持`
- 辅助说明：`如果你已经通过其他方式支持过这只动物，可以在这里补登记，等待救助人确认。`

## 8.3 支持登记串

#### 支持登记串（Support Thread）

表示：

**一个支持者对一只动物的支持关系。**

约束：

- `(animal_id, supporter_user_id)` 唯一

作用：

- 把同一个人多次支持同一只动物的记录串起来
- 防止系统把多次支持误当成多个独立“人”
- 方便救助人按“人”核对，而不是按零散记录核对

#### 支持记录（Support Entry）

表示：

**该支持串里的一次具体支持。**

例如：

- 4/1 支持 200
- 4/2 又支持 300

这两条都存在，但属于同一个支持串。

## 8.4 记录状态

顶层状态：

- `待处理`
- `已确认`
- `未匹配`

说明：

- `待处理`：等待救助人核实
- `已确认`：核实成立，计入已确认支持金额
- `未匹配`：当前没匹配上，不代表系统判定用户造假

## 8.5 未匹配原因

一期先做枚举原因：

- 未查到对应转账
- 金额或时间不匹配
- 截图信息不足
- 疑似重复提交
- 疑似无关记录
- 其他

这里的“重复”作为未匹配原因，而不是顶层状态。

## 8.6 关键约束

### 1. 支持记录不能删除

- 支持者提交后，记录必须保留
- 救助人不能删除
- 只能确认或标记未匹配

### 2. 未匹配不计入已确认支持

前台金额只统计：

- 所有 `已确认` 记录金额之和

`待处理 / 未匹配` 不计入已收到金额。

### 3. 未匹配一期不直接影响首页权重

MVP 先只展示，不做自动降权。

### 4. 轻量反刷限制

一期建议加 3 条：

- 同一用户对同一动物，10 分钟内最多新增 1 条支持记录
- 同一用户对同一动物，24 小时内最多新增 3 条支持记录
- 同一支持串内，相同截图不能重复提交

### 5. 重复提交要能被保留但不重复入账

如果一条记录被标记为“疑似重复提交”：

- 保留记录
- 不重复计入金额
- 在支持者侧可见当前处理结果

## 8.7 救助人端处理方式

救助页和单只动物页只显示轻通知：

- 有 2 条待处理的支持登记

进入后按支持串展示：

- 支持者昵称
- 累计已确认金额
- 待处理条数
- 最新一条记录时间

展开后看具体支持记录：

- 4/1 ¥200 待处理
- 4/2 ¥300 已确认
- 4/3 ¥100 未匹配

救助人对单条记录可执行：

- `确认收到`
- `暂未匹配`

“重复”通过未匹配原因表达，不单独做第三个主按钮。

## 9. 这一版最值得马上采纳的改动

1. 首页卡片固定展示 `最新情况 + 资金状态 + 推荐理由`
2. 把首页资格状态做成白话可行动文案
3. 把 `公开转发` 和 `首页展示` 继续分层，但表达更清楚
4. 保留动物中心化工作流，不回到重工作台
5. 把公共证据机制写进统一支出模板，而不是只靠批量记账解释
6. 个人中心只保留“支持足迹”，且只有用户提交过“我已支持”后才出现内容
7. 保留极简救助人主页，但只做案例卡片列表
8. 资金表达改成“总预算 + 已确认支出 + 已确认支持 + 当前垫付状态提示”
9. 继续坚持一期不默认前置公开收款码

## 10. 一句话总结

v3.2 不再试图“多做一点”，而是把真正决定成败的几个结构定死：

- 首页要能判断
- 个案页要能说服
- 记账要真的记得下去
- 支持闭环要现实可信
