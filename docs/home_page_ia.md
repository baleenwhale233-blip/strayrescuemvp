# 首页 / 发现 页面 IA

## 1. 页面定位

首页不是案例流，而是：

**帮助支持者快速判断现在先看谁。**

它的成功标准不是停留时长，而是用户能不能在 30-60 秒内完成：

- 找到一个值得先点进去的个案
- 理解这只为什么被放在前面
- 愿意进入透明账本页继续判断

### 1.1 首页准入规则（一期当前执行口径）

首页不是“所有公开案例”的集合页。

当前一期里，只有满足下面 3 条的案例，才会进入首页列表：

1. 案例已经公开，也就是 `visibility = published`
2. 至少有 1 条公开的进展更新
3. 至少有 1 条支出记录，且当前记录凭证情况不是 `待补充`

也就是说，当前首页资格看的不是“谁更惨”或“谁预算写得更高”，而是：

- 是否已经公开
- 是否持续更新
- 是否已经开始形成基础透明记录

如果不满足，会回到几条固定的人话原因：

- `未公开，暂不进入首页`
- `还缺一条最近更新`
- `还要补一条支出凭证`
- `已满足首页条件`

当前还**没有**把下面这些做成首页拦截条件：

- 预算写得多不多
- 支持金额大不大
- 支持次数多不多
- 异常 / 重复风险分

这些可以作为后续风控或排序增强，但不是一期首页准入的当前执行口径。

---

## 2. 页面结构

```text
发现
  A. 顶部说明
  B. 案例 ID 搜索入口
  C. 主筛选条
  D. 记录列表
  E. 判断说明入口
```

### 模块 A：顶部说明

一句短文案：

`这里展示的是符合平台首页条件、适合优先判断和支持的个案。`

不要讲太多规则，只定用户预期。

### 模块 B：案例 ID 搜索入口

首页顶部增加一个轻量搜索入口。

定位不是“全站搜索”，而是：

**外部平台导流后的查档入口。**

#### 一期支持内容

- 只支持案例 ID 精确搜索
- 支持输入：
  - `482731`
  - `JM482731`

#### 命中行为

- 精确命中后，直接跳个案详情页
- 不进入搜索结果页

#### 未命中行为

- 只给轻提示
- 示例：`没有找到这个案例 ID`

#### 一期不做

- 不做动物名字搜索
- 不做模糊搜索
- 不做筛选组合搜索

#### 案例 ID 规则

每个公开案例生成一个唯一、可传播、适合手打的短 ID。

建议格式：

- 展示值：`JM482731`
- 搜索兼容：`482731` / `JM482731`

### 模块 C：主筛选条

主筛选项：

- 已垫付较多
- 记录凭证较齐
- 刚更新
- 快完成

这里不保留“最缺口”。

原因是一期缺口里混有预算成分，容易被虚高预算污染。相比之下，“已确认支出形成的垫付压力”比“预测缺口”更适合作为首页判断维度。

辅助筛选进入折叠区：

- 医疗中
- 康复中
- 待安置
- 已完成

### 模块 D：记录列表

一期不单独做“平台推荐位”。

首页直接展示：

**所有有资格进入首页的案例卡片，并按最近更新时间倒序排序。**

这里的“有资格进入首页”，当前就对应上一节的 3 条执行口径，不再额外引入隐藏分数。

列表卡的任务不是讲完整故事，而是促成点击。

#### 点击进入详情

- 从发现页点击案例时，只携带案例 `id`。
- 详情页先尝试读取当前用户的 owner 详情；如果当前用户就是该案例记录维护者，直接展示主态详情。
- 如果不是 owner 或 owner 详情不可用，则展示客态详情，保持外部查看者查档路径稳定。

#### 信息顺序

1. 动物基础信息
2. 最新情况
3. 资金状态
4. 推荐理由
5. 记录凭证情况
6. 最近更新时间

其中状态标签的 emoji 口径应与“写进展更新”页保持一致：紧急送医 `🚨`、医疗救助中 `🏥`、康复观察 `🏡`、寻找领养 `💖`、遗憾离世 `🌈`。

#### 三个摘要位的来源

- 最新情况：救助人填写
- 资金状态：系统计算
- 推荐理由：系统输出

#### 资金状态文案规则

首页卡片里的“资金状态”不直接展示复杂分段语义，统一回答：

**已确认支持离总预算还有多远。**

这里的 `target_amount` 指预算调整后的当前总预算；如果 bundle 里同时有建档预算和预算调整记录，系统以当前总预算计算资金状态。

1. 如果 `target_amount <= 0`
   - 显示：`预算待确认`
2. 如果 `confirmed_support_amount >= target_amount`
   - 显示：`已达总预算`
3. 如果 `0 < target_amount - confirmed_support_amount <= 300`
   - 显示：`即将筹满`
4. 其他情况
   - 显示：`距离预算还差较多`

这里的筹满判断只基于：

- 总预算
- 已确认支持

不再用 `已确认支出 - 已确认支持` 判断“即将筹满”。

#### 资金进度条口径

首页卡片里的资金条总长度直接代表 `target_amount / 总预算`。

- 橙色填充只代表 `confirmed_support_amount / target_amount`
- 救助人当前垫付金额只作为数字指标和推荐理由依据展示
- 不再把“救助人已垫付 / 当前差额”画进进度条分段里

这样用户看到的条是一把预算尺，而不是“已花 + 已支持”的对比条。

#### 推荐理由规则

一期固定只做 4 类规则，从上到下命中第一条就显示，不同时展示多条：

1. 垫付压力型
   - `已垫付 ¥1800，仍待补位`
2. 刚更新仍缺口型
   - `刚更新病情，当前仍缺 ¥420`
3. 接近完成型
   - `接近完成，只差最后 ¥300`
4. 记录凭证较齐型
   - `记录和凭证较齐，仍有缺口`

#### 字段条件与文案模板映射

首页推荐理由不使用大模型临场生成。

一期采用：

- 固定字段
- 固定计算口径
- 固定命中顺序
- 固定文案模板

##### A. 首页依赖的基础字段

###### 救助人填写

- `latest_status_summary`
  - 最近一次更新时填写的卡片摘要短句
- `latest_status_updated_at`
  - 最近一次公开状态更新的时间

###### 系统计算

- `confirmed_expense_amount`
  - 已确认支出总额
  - 口径：所有已确认支出记录金额之和
- `confirmed_support_amount`
  - 已确认支持总额
  - 口径：所有已确认支持金额之和
- `budget_gap_amount`
  - 当前预算差额
  - 计算：`max(target_amount - confirmed_support_amount, 0)`
- `advance_gap_amount`
  - 当前垫付差额
  - 计算：`max(confirmed_expense_amount - confirmed_support_amount, 0)`
- `case_evidence_level`
  - 个案当前记录凭证情况
  - 取值：`完整 / 基础 / 待补充`
- `last_public_activity_at`
  - 最近一次公开活动时间
  - 口径：最近一次公开状态更新、支出记录或支持确认的时间

##### B. 一期推荐理由命中顺序

从上到下判断，命中第一条就停止：

1. 垫付压力型
2. 刚更新仍缺口型
3. 接近完成型
4. 记录凭证较齐型
5. 都未命中则不显示推荐理由

##### C. 条件与模板

###### 1. 垫付压力型

**命中条件**

- `confirmed_expense_amount >= 1000`
- 且 `advance_gap_amount >= 500`

**文案模板**

- `已垫付 ¥{confirmed_expense_amount}，仍待补位`

**说明**

- 这一类强调的是“已经真实发生、且还没补回”的压力
- 不用它判断“筹满进度”

###### 2. 刚更新仍缺口型

**命中条件**

- 最近 48 小时内有公开更新
- 且 `budget_gap_amount > 0`

**文案模板**

- `刚更新病情，当前仍缺 ¥{budget_gap_amount}`

**说明**

- 这条强调“人在持续推进，不是失联个案”
- 这里的“仍缺”指距离总预算仍缺，不再用垫付差额

###### 3. 接近完成型

**命中条件**

- `0 < budget_gap_amount <= 300`

**文案模板**

- `接近完成，只差最后 ¥{budget_gap_amount}`

**说明**

- 这里的“接近完成”不是指整案治疗结束
- 只表示已确认支持离总预算只差最后一小段，适合补最后一段

###### 4. 记录凭证较齐型

**命中条件**

- `case_evidence_level = 完整`
- 且 `budget_gap_amount > 0`

**文案模板**

- `记录和凭证较齐，仍有缺口`

**说明**

- 这条适合那些没有明显“垫付压力”或“快完成”信号，但可信度已经足够高的个案

##### D. 显示规则

- 一张卡最多显示 1 条推荐理由
- 推荐理由一定是系统判断，不和“最新情况”混写
- 推荐理由不解释全部故事，只回答“为什么值得现在点进去”
- 所有金额都显示为整数元或标准人民币格式，例如 `¥420`

##### E. 一期不做的事

- 不做“预算越高越优先”的推荐理由
- 不做“倒计时 / 截止时间”类推荐理由
- 不做多条推荐理由叠加
- 不做模型自由生成文案

#### 首页状态文案 vs 推荐理由

两者不要混写：

- `资金状态`
  - 回答：已确认支持离总预算还有多远
- `推荐理由`
  - 回答：为什么这只值得现在点进去

例如：

- 资金状态：`距离预算还差较多`
- 推荐理由：`记录和凭证较齐，仍有缺口`

#### 交互方式

- 不单独放 CTA
- 默认整卡点击进入详情页
- 首页保持“查档 + 判断”感，不再额外加操作按钮

### 模块 E：判断说明入口

轻入口文案：

`我们如何把个案放到首页`

展开后说明：

- 首页不是按惨烈程度排序
- 也不是按预算写得多来排序
- 公开页不等于首页展示
- 当前执行口径优先看持续更新和基础支出凭证
- 当前预算差额、垫付压力和闭环可能性主要体现在卡片推荐理由里，不单独做首页拦截分

---

## 3. 关键状态

### 记录凭证情况

- 完整
- 基础
- 待补充

### 首页推荐理由

必须始终是系统判断，不和最新情况混写。

### 案例 ID

- 每个公开案例必须有一个短 ID
- 短 ID 必须稳定、唯一、适合手打
- 首页搜索只认这个 ID

### 空状态

如果首页暂时没有符合条件的个案：

- 不要显示“暂无数据”
- 改成：
  - `当前没有符合首页展示条件的个案`
  - `你仍然可以通过分享进入公开页查看正在传播的救助个案`

---

## 4. 页面原则

1. 不做沉浸式刷流
2. 不靠大惨图驱动点击
3. 一屏内必须出现“为什么先看这只”
4. 不做首页推荐位，先把案例列表的信息结构做稳
